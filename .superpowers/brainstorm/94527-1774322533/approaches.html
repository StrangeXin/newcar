三种实施方案对比

全面升级（Token + 组件 + 布局），三种推进方式各有侧重

A

Token 优先 · 由内而外

第一步建完整的设计 token 系统（颜色阶梯、字型比例、间距、阴影、圆角），同时集成字体加载和 data-theme 主题切换机制。Token 稳定后，再系统地升级全局组件（按钮、卡片、徽章、导航），最后重构各页面布局。

优点

  • 主题切换最干净，零硬编码颜色
  • 后续改动只改 token，不碰组件
  • 两套主题天然一致

代价

  • 前期工作量大，视觉效果出来慢
  • 需要先完整规划 token 命名体系
B

页面优先 · 逐页重做

按页面重要性排序（首页 → 旅程工作台 → 社区广场 → 聊天），每个页面从视觉整体重新设计，设计模式自然沉淀为组件。主题切换在全部页面升级完成后再统一提炼。

优点

  • 最快看到完整页面效果
  • 可以分批上线，每个页面独立

代价

  • 页面间容易出现风格漂移
  • 主题切换需要最后补做，成本高
C（推荐）

双轨并行 · Token + 核心组件同步

分三轨推进：① 精简版 token 系统（只定义使用最频繁的变量，够用即可）+ 字体 + 主题切换基础架构；② 核心组件库升级（Button、Card、Badge、ChatBubble、FeedCard 等 ~10 个组件）；③ 三个主要页面布局重构（首页 Hero、工作台三栏、社区 Feed）。三轨有依赖关系但可快速推进。

优点

  • 平衡系统性和交付速度
  • 主题切换从第一天就是正确的
  • 组件复用率高，改动集中

代价

  • 需要同时管理三个工作流
  • token 覆盖度不如方案 A 彻底

我推荐 C — 在可用的时间内交付最大视觉提升，同时保证主题系统从第一天就是对的。在终端确认。
